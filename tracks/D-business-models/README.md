# Track D — ビジネスモデルの選択肢列挙

## 前提

ノンカストディ型・自己責任型を選ぶとした場合(Track B 結果で確定)、収益化モデルは限定される。

## 5 候補

### 1. SDK ライセンス
- 形態: B2B のみ
- 顧客: 決済プロセッサ・大手 PSP・AI agent マーケットプレイス
- 価格帯: 200-1,000 万円/社/年
- 検証項目: 顧客候補の SDK 採用意思、価格受容、競合との差別化

### 2. Hosted Audit/Dispute Backend
- 形態: SaaS
- 価格帯: 月100-500 万円/顧客
- **重要な制約**: コタ側がデータ預かりライアビリティを負う → ノンカストディ型の利点が薄れる
- 検証項目: そもそもノンカスト原則と両立可能か

### 3. Conformance Test / Certification
- 形態: 規格認証ビジネス
- 検証項目: 標準化団体との連携可否、市場規模、収益性

### 4. Insurance Layer 代理店
- 形態: 手数料収入
- パートナー: Lloyds、国内損保
- 検証項目: Agent Payments Insurance 商品の組成可能性、再保険市場の関心

### 5. Hybrid: OSS コア + 商用エンタープライズ機能
- 形態: GitLab / Confluent 型
- 検証項目: コミュニティ形成、商用機能の差分設計、収益化までの時間

## 進め方

1. **W1-W2**: 5モデル定義、評価フレームワーク確定
2. **W3-W4**: モデル別経済性試算(ARR / マージン / 必要顧客数)
3. **W5-W6**: 顧客インタビュー仮説(誰に何を聞くか)
4. **W7-W8**: Track A の競合モデル分布と統合
5. **W9-W10**: 顧客3社特定(decision-doc.md §4 に反映)
6. **W11**: 最終モデル選定(1-2 つ)→ decision-doc.md §2 に反映

## 評価軸

- 収益性(ARR、マージン、scaling 性)
- ノンカストディ原則との整合性(2 が要注意)
- 顧客獲得難度
- 競合度合い(既存プレイヤーとの差別化)
- Phase 1 の実装負荷
- PayPay との関係(Path A/B/C)で取りやすいモデル
